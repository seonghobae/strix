# Password hashing: never SHA-256

This repo treats the following security finding as a common, high-signal issue:

> Sensitive data (password) is used in a hashing algorithm (SHA256) that is
> insecure for password hashing, since it is not a computationally expensive
> hash function.

## Why SHA-256 is wrong for passwords

SHA-256 is a **fast** hash. Fast hashes make offline cracking cheaper and faster
after a credential database leak. Password storage requires a **slow**,
parameterized password hash (KDF) to increase attacker cost.

## First: confirm it is actually password hashing

Some scanners produce false positives when `SHA256` appears near the word
"password".

Treat the finding as a hypothesis until confirmed:

- **Source**: does the input originate from a password field (registration,
  login, reset, change-password flows)?
- **Sink**: is the digest stored in a user credential store
  (`users.password_hash`, `credentials.hash`, etc.) or used to verify login?
- **False positive examples (not password hashing)**:
  - file checksums / integrity
  - ETags/cache keys
  - hashing high-entropy random tokens for storage/lookup (still use
    constant-time compare)

If it's not password hashing, document the evidence and (optionally) narrow the
scanner scope to avoid future noise.

## Required remediation (default)

### Algorithm choices

1. **Argon2id** (preferred)
- Store the output as a **PHC string** (self-describing: algorithm + params +
  salt + hash).

Practical baseline parameters (tune with perf tests):
- memory: **64 MiB** (`m=65536` KiB)
- time cost: **t=3**
- parallelism: **p=1**
- salt: **16 bytes** cryptographically secure random (use a CSPRNG) per password
- hash length: **32 bytes**

1. **bcrypt** (acceptable)
- cost/work factor: **12–14** (start at 12, tune with perf tests)
- note: bcrypt effectively truncates long inputs (~72 bytes). Prefer Argon2id if
  long passphrases are supported.
- salt: generated by the bcrypt library (MUST use a CSPRNG; do not roll your own)

1. **scrypt** (acceptable)
- baseline: `N=32768, r=8, p=1`, salt 16 bytes (CSPRNG), keyLen 32

### Salt generation (CSPRNG required)

Salts MUST be generated using a cryptographically secure pseudorandom number
generator (CSPRNG).

Examples:

- Python: `secrets.token_bytes(16)` (or `os.urandom(16)`)
- Node.js: `crypto.randomBytes(16)`
- Go: `crypto/rand.Read(buf)`
- Java: `SecureRandom` (e.g., `new SecureRandom().nextBytes(buf)`)

### Storage format

The stored password hash MUST be self-describing so the verifier can branch by
prefix/metadata:

- Argon2id PHC example:
  - `$argon2id$v=19$m=65536,t=3,p=1$<b64salt>$<b64hash>`
- bcrypt example:
  - `$2b$12$...`

Ensure the DB field length can store these strings safely (typically
`VARCHAR(255)` or larger).

### Constant-time verification

Prefer library-provided `verify/compare` APIs. If you must compare manually,
use constant-time functions:

- Python: `hmac.compare_digest`
- Node: `crypto.timingSafeEqual`
- Go: `crypto/subtle.ConstantTimeCompare`
- Java: `MessageDigest.isEqual`

## Migration strategy (no forced reset by default)

Default approach: **verify legacy → rehash on login**.

1. Support verifying both formats:
   - new format (Argon2id/bcrypt/scrypt)
   - legacy SHA-256 format (only for verifying existing users)
1. On successful login with legacy SHA-256:
   - immediately rehash with the new algorithm and update the stored value
1. Password reset/change MUST write only the new format.

After an adoption window, you may drop legacy verification only if you have an
approved plan for remaining legacy accounts.

### Migration monitoring (recommended)

Track progress so you can plan (or avoid) legacy format deprecation:

- Count upgrades: `auth_password_hash_upgrades_total{from="sha256",to="argon2id"}`
  (or equivalent for bcrypt/scrypt).
- Measure verification latency: `auth_password_verify_duration_seconds`
  (histogram) to catch regressions.
- Track remaining legacy users (gauge): e.g., `users_with_legacy_password_hash`.
- Ensure logs/metrics never include passwords, raw hashes, salts, or peppers.

## Minimal mechanical gates (tests)

Add tests that prove behavior and prevent regressions:

1. New-hash roundtrip
- `verify(hash(pw), pw) == true`
- `verify(hash(pw), wrong) == false`

1. Salt uniqueness
- hashing the same password twice produces different stored strings

1. Migration upgrade
- legacy SHA-256 stored hash can login
- successful legacy login upgrades stored hash to Argon2id/bcrypt/scrypt
- subsequent login works using the new format

## Closure note (for PR/security finding)

"SHA-256 is a fast hash and is unsuitable for password storage. We replaced
password hashing/verification with a slow password hash (Argon2id preferred;
fallback bcrypt/scrypt) using a self-describing stored format. Existing SHA-256
hashes are supported for verification only and are upgraded on successful login
(verify-then-rehash). Tests cover roundtrip verification, salt uniqueness, and
legacy upgrade behavior."
