# DDD Guidelines (Agent Default)

이 문서는 **에이전트 작업 기본값으로 DDD를 적용**하는 기준을 정의한다.
이 저장소는 구성/정책 중심이므로, 실제 도메인 모델링은 **다운스트림 서비스 레포**에서 수행한다.

## 언제 적용하는가

- 도메인 규칙/정책이 복잡하거나 자주 변하는 경우
- 용어 혼동으로 변경 위험이 큰 경우(같은 단어가 다른 의미)
- 다중 팀/서비스 경계에서 개념 충돌이 발생하는 경우

### 적용하지 않아도 되는 경우(예외)

- 단순 CRUD 수준이며 규칙이 거의 없는 경우
- 짧은 수명/프로토타입 작업

## 핵심 개념

### 유비쿼터스 언어 (Ubiquitous Language)
- 도메인 용어를 코드/테스트/문서에 **동일하게** 사용한다.
- 모호한 용어는 설계 결함으로 보고 용어를 정리한다.

### 바운디드 컨텍스트 (Bounded Context)
- 모델이 일관된 경계(용어/규칙이 내부에서 충돌하지 않음).
- 컨텍스트 간 통합 시 컨텍스트 맵을 명시한다.

### 엔티티 / 값 객체 / 애그리게이트
- **Entity**: ID 중심, 시간이 지나도 동일 정체성 유지
- **Value Object**: ID 없음, 값 동일성, 불변(immutable)
- **Aggregate**: 일관성 경계. **Aggregate Root만 외부 노출**

### 도메인 서비스 (Domain Service)
- 엔티티/값 객체에 속하지 않는 순수 도메인 로직
- I/O 없음, 테스트 용이해야 함

### 리포지토리 (Repository)
- Aggregate Root 접근 인터페이스
- 구현은 인프라 계층에 위치

## TDD + DDD 결합 방식(기본값)

1. **유비쿼터스 언어 정리**: 용어/모델을 테스트 이름과 코드에 먼저 반영
2. **도메인 레이어 테스트 작성(RED)**: 규칙/불변성/상태전이를 실패 테스트로 명시
3. **최소 구현(GREEN)**: 엔티티/값 객체/애그리게이트 규칙을 최소 구현
4. **리팩터링**: 중복 제거, 용어 정렬
5. **리포지토리/통합 테스트**: 저장소/경계 매핑 확인

## 작업 체크리스트(에이전트용)

- [ ] 용어 정리(테스트/코드/문서 동기화)
- [ ] Aggregate Root 경계 밖 직접 참조 금지
- [ ] 도메인 서비스는 순수 로직으로 유지
- [ ] 테스트 우선(TDD) 단계 기록
